# 数仓岗位100问

  • 编程语言方面要求一定要掌握java吗？

    目前没见过要求一定要掌握java的数仓岗位，但是平台开发都要求java

  • 日常工作

    基本就是业务理解 需求沟通 然后开发数仓SQL任务

    （一个需求 要好多天 ，前期业务理解 数据调研 评审 写方案 占了六七成； 写代码占两三成 ；还有些上线 的操作）

  • 如何从0搭建数据仓库

    这个问题回答起来其实就是数据仓库的意义以及分层定义，比如数据规范化，集成，聚合，抽取主题等。说的太笼统，因为不同产品形态，不同数据体量，甚至不同阶段，对于数据仓库的重点都不同。阿里有一批人做数据抽样，按道理大数据就是不做抽样做整体。但是当数据量大到一定程度，不做抽样就没办法做快速查询。

  • 数据仓库岗位的天花板在哪里

    所有的技术非管理，级别最高是架构师。目前从招聘来看，阿里的7，美团的32，都是可以只做数据仓库。再往上就不单单是技术了

  • 如何衡量数仓做的好坏

    • A：这个问题难回答。我觉得可以用数据使用情况以及需求响应时间，计算节省时间等指标，但是还是不好评分。面试阿里的时候被问到这个问题。面试官让我做一个评分体系对数据仓库。我的回答是，一般的想法是看表与字段的使用情况比如使用率来评价，但是数据仓库除了满足现有需求之外，还要做一些提前设计（根据业内情况，产品改变等）所以不能用表，字段的使用率等指标简单的评价。因为如果为了评分高，我就不做提前设计等需求来再调整，使用率一定高。如果加上需求反馈情况作为评判标准，这个权重又不好控制。

      另外计算效率的提升，只有初期的时候才有和直接计算的对比，等全面使用数据仓库了就没有对比组了。请问您有什么好方法

    • B（回复A）：衡量数仓即是衡量数仓模型，可以从业务和技术的角度去做一些量化指标。如你所说，数据热度，对业务的覆盖度，计算资源和存储的量化，响应的量化。数据仓库做到后期就是数据资产，你说的那个面阿里的评分应该就是数据资产的评分体系。具体我也没落地过，只是之前看到过同事的关于这块的文章

    • A（回复B）：其实就是落地难，评分的规则，比较好整理，但是怎么权衡，以及评分之后的后续不好办。评分低的要下线可能是一个思路但是实现起来也有难度。

      并且在离业务较远的表，评分会低，但是它是基础。我也想过，但是绕来绕去没想到好方法，就没继续研究

  • 数仓的意义在哪里？之前在互联网公司，裁员的时候首先是bi部门

    数据仓库的意义是，让公司从数据层面知道产品的好坏，功能的好坏，往大了说，辅助公司做决策。

    产品做的好不好取决于产品本身而不是技术。数据是锦上添花，做不到雪中送炭。微信的微视做不好绝对不是技术能力不够。

  • 数仓如果从组织架构上分成公共数仓和业务数仓，有什么好处和坏处？

    分成公共数仓和业务数仓，对公司是好的，对员工尤其是公共数仓的员工不好。划分开之后，公司有统一的数据规范，统一的规范化，各业务也能灵活组织数据，但是对于公共数仓的同学，完全脱离业务，数据就是数据，不知道也没必要知道含义，长期下去，会觉得没有成长。当然我说的是数据层面的成长，其它方面比如增加工具引入新架构，还是有很多可做的

  • 数仓怎么更好的贴近业务，或者怎样使数仓扩大影响力？

    贴近业务已经是答案了，多了解自己产品的变化，做好沟通，知道要做什么改变为什么改变想达到什么效果。不仅仅做需求，适当做一些提前设计，可以参考业内其它同类产品的发展来思考。总之就是，多沟通多了解。

  • 数仓怎么实现精细化运营

    精细化运营是数据的使用，数仓是基础，做好基础至于怎么用，运营的同学更懂。

  • 数仓的职业发展路径是怎样的？相对于后端开发哪个前景更好呢？

    这个太宏观不好回答，姑妄言之。先说发展前景，没什么好坏之分。路径，先技术后业务，贴近业务才能做好数据

  • 那一开始就做业务数仓是不是不太好呀，应届生你是推荐业务数仓还是JAVA呢？

    不敢推荐，毕竟这个影响人的一生。个人感觉，java比较平稳，以后转也容易。如果做数据，关注业务，同时也不能无视工具，我们是使用工具的，但是也要知道工具工作原理，这样在做优化时，更能准确的解决问题

  • 请教一下，数仓这个岗位还会存在多久？核心竞争力是什么？本人应届硕士数仓，感觉业务比不过BI,BA，代码比不过数据平台和后端，不知道出路在哪

    岗位一直会存在。核心竞争力是服务能力。1，保证能准确提供数据，2，快速准确提供数据，先做到这些，再想后续发展就行。

    这个行业不会没有，现在都是用数据说话，但是一些业务，或者运营的同学没有数据基础，还是需要有人整理数据，根据他们的需求提供数据支持，所以不要太担心。
