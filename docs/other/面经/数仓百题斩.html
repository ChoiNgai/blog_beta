# 数仓面试题汇总

# 1. 维表和宽表的考查（主要考察维表的使用及维度退化手法）

维度表：数据仓库中的表，其字段全是描述事实数据表中的数据（比如日期、设备类型是安卓还是ios还是PC还是ipad等），维度表包含创建维度所基于的数据。

事实表：也叫事实数据表，就是app埋点日志，主要特点是含有大量的数据，并且这些数据是可以汇总，并被记录的。

# 2.数仓命名规范

这里分为表命名规范、字段命名规范、指标命名规范、作业命名规范，通常问表命名规范

  • 表命名规范

    不同分层的表命名规范不一样（以下命名规范包括了库名），abc为补充的三个参数，具体见后面

    • ODS层

      即贴源层，由业务方直接导入，不会经过数据清洗转化。操作数据层定位于业务明细数据保留区，负责保留数据接入时点后历史变更数据，数据原则上全量保留。

      项目名_ods.ods_系统名称(默认为空)_源表名_abc

    • DW层

      由ODS层数据经过清洗转化而成。这一层整合后的业务过程明细数据，负责各业务场景整合、数据结构化、规范化，常用公共维度冗余加工，以及明细业务标签信息加工。另外，在该层也会做一部分的数据聚合，将相同主题的数据汇集到一张表中，提高数据的可用性。

      • dim层（维表）

        项目名_dw.dim_项目名_业务描述_abc

      • dwv（业务明细）

        项目名_dw.dwv_项目名_一级主题域_二级主题域_业务描述_abc

      • dwm（周期汇总）

        项目名_dw.dws_项目名_数据粒度_业务描述_统计范围cycle[N]_abc

        例如：

        projectname_dw.dwm_projectname_store_activity_byr_currcy_order_cycle7_sdt说明：projectname是项目名，其数据粒度为店铺（store）+活动（activity）+买家（byr）+币种（currcy）;业务描述为下单（order）；统计的时间范围为近7天。

      • dws层（数据汇总服务）

        项目名_dw.dws_项目名_数据粒度_业务描述_abc

    abc 说明

    a：表示数据粒度类别: f 表示明细数据，s表示汇总数据。

    b：表示数据更新周期: d表示天(t+1)产出; h表示小时产出; mi表示分钟产出; c表示全量产出; w表示周产出; m表示月产出，y表示月产出。

    c：表示数据存储形态: t表示表; v表示视图。

    abc会有多种不同的组合，比如 fct 表示dw层的全量表，fdt 表示dw层的增量表(天表)

  • 作业命名规范

    • 数据采集作业（后缀import）

    一般是采集变成ods层数据

    项目名_ods.ods_系统名称_源表名_abc_import

    如：projectname_ods.ods_projectname_orders_fct_import

    • 数据导出作业（后缀export）

      同采集作业，但后缀改成export

      shopline_store_method_h_export

    • 数据查询作业

      与产出目标表同名

  • 字段命名规范

    • 度量词汇

      次数：cnt

      金额：amt

      时长：dr

      比例：rate

      百分比：pct

      环比：p2p

    • 统计词汇

      订单金额：order_amt

      商品数：prod_cnt

      用户数：uid_cnt

      订单数：order_cnt

      曝光次数：pv

      点击次数：click_cnt

      买家数：byr_cnt

      卖家数：sel_cnt

      店铺数：store_cnt

    • 约定名词词汇

      gmv：成交金额

      cmv：取消金额

      pmv：营业额

      cancel：取消

      consumer：顾客（有消费记录）

      user：用户

      member：会员

      coins：购物金

      sent：发送

      used：使用

      unuse：待使用/未使用（不含已过期）

      source：来源

      promotion：优惠

    • 日期修饰词汇

      日：d

      小时：h

      分钟：mi

      周：w

      月：m

      季度：q

      年：y

      七日：7d

    • 聚合词汇

      total：历史累计

      avg：平均

      std：截止到当前统计日期

    • 业务场景词汇

      order：下单

      pay：支付

      refund：退款

      return_goods：退货

      subs：订阅

      reg：注册

      login：登录

      visit：访问

      add_chart：购物车

      site：站点

  • 指标命名规范

    1、指标统一：数据指标的内容及对应的命名、属性需要保持唯一性，无歧义。

    2、指标统一目前只针对单一业务，暂不考虑跨业务间场景

    3、指标统一命名：修饰语（业务场景限定条件）+对象+方法+时间范围 如cancel_order_cnt_1d

    修饰语（业务场景限定条件）：cancel 取消；

    统计对象：order 订单；

    统计方法：cnt 记数，对订单进行去重统计；

    统计周期：1d近一天的。

    4、原子、派生指标之间的定义和关系

    ​ 原子指标：基于某一业务事件行为下的度量，是业务定义中不可再拆分的指标，具有明确业务含义的名词 ，如支付金额。

    ​ 派生指标: 由原子指标＋多个修饰词（可选）＋时间周期。可以理解为对原子指标业务统计范围的圈定。如原子指标：支付金额，最近一天海外买家支付金额则为派生指标（最近一天为时间周期，海外为修饰词 买家作为维度，而不作为修饰词）

# 4.数据库和数据仓库有什么区别

  • 数据库是面向事务的，数据由日常业务产生，常更新

    数据仓库是面向主题的，数据来源于数据库或文件，经过一定的规则转换得到，用于分析决策

  • 数据库一般是存储当前数据的

    数据仓库一般存储的是历史数据

  • 数据库的设计一般要符合三范式，有最大精确度和最小冗余度，有利于数据的插入

    数据仓库的设计一般是按照数仓建模规范，有一定的冗余，有利于查询

# 45.告警平台你们怎么做的，说一下

我说一下告警方案，比如埋点日志上报，像是用户ID这个字段是必不可少的，这种情况就通常直接计算他的空值个数，实现方式：

SELECT COUNT( IF(userid IS NULL,1,0) ) AS failnum FROM tablename

一旦大于0就邮件+短信告警

而有些字段是可以允许出现那么点错误的，比如网页曝光成功率（用户量大的时候肯定没办法保证每个用户每次都成功地打开了业务），这种情况就可以设定如果低于某个阈值就告警，实现方式：

假设oss_status =1 表示访问成功，oss_status =2 表示访问失败，oss_status =3表示其他情况

SELECT COUNT( IF(userid IS NULL,1,0) )/COUNT(*) AS p FROM tablename

# 47.数据模型如何构建，星型、雪花、星座的区别和工作中如何使用；

数仓建模中一般用星型模型，而雪花模型在关系型数据库中比较常用，因为雪花模型的各个表冗余较低，需要比较多Join，Join操作MapReduce中严重影响性能。
